43 // Base class of statistics algorithm class hierarchy.
44 
45 // The default implementation is such that statistics are only calculated when
46 // getStatistic() or getStatistics() is called. Until then, the iterators which
47 // point to the beginning of data sets, masks, etc. are held in memory. Thus,
48 // the caller must keep all data sets available for the statistics object until
49 // these methods are called, and of course, if the actual data values are changed
50 // between adding data and calculating statistics, the updated values are used when
51 // calculating statistics. Derived classes may override this behavior.
52 //
53 // PRECISION CONSIDERATIONS
54 // Many statistics are computed via accumulators. This can lead to precision issues,
55 // especially for large datasets. For this reason, it is highly recommended that the
56 // data type one uses as the AccumType be of higher precision, if possible, than the
57 // data type pointed to by input iterator. So for example, if one has a data set of
58 // Float values (to which the InputIterator type points to), then one should use type
59 // Double for the AccumType. In this case, the Float data values will be converted to
60 // Doubles before they are accumulated.
61 //
62 // METHODS OF PROVIDING DATA
63 // Data may be provided in one of two mutually exclusive ways. The first way is
64 // simpler, and that is to use the setData()/addData() methods. Calling setData() will
65 // clear any previous data that was added via these methods or via a data provider (see
66 // below). Calling addData() subsequently to setData() will add a data set to the set
67 // of data sets on which statistics will be calculated. In order for this to work
68 // correctly, the iterators which are passed into these methods must still be valid when
69 // statistics are calculated (although note that some derived classes allow certain
70 // statistics to be updated as data sets are added via these methods. See specific
71 // classes for details).
72 //
73 // The second way to provide data is via a data provider. This takes the form of
74 // a derived class of StatsDataProvider, in which various methods are implemented for
75 // retrieving various information about the data sets. Such an interface is necessary for
76 // data which does not easily lend itself to be provided via the setData()/addData()
77 // methods. For example, in the case of iterating through a lattice, a lattice iterator
78 // will overwrite the memory location of the previous chunk of data with the current chunk
79 // of data. Therefore, if one does not wish to load data from the entire lattice into
80 // memory (which is why LatticeIterator was designed in this way), one must the
81 // LatticeStatsDataProvider class, which the statistics framework will use to iteratate
82 // through the lattice, only keeping one chunk of the data of the lattice in memory at one
83 // time.
84 //
85 // QUANTILES
86 // A quantile is a value contained in a data set, such that, it has a zero-based
87 // index of ceil(q*n)-1 in the equivalent ordered dataset, where 0 < q < 1 specifies
88 // the fractional location within the ordered dataset and n is the total number of elements.
89 // Note that, for a dataset with an odd number of elements, the median is the same as
90 // the quantile value when q = 0.5. However, there is no such correspondance between the
91 // median in a dataset with an even number of elements, since the median in that case is
92 // given by the mean of the elements of zero-based indeces n/2-1 and n/2 in the equivalent
93 // ordered dataset. Thus, in the case of a dataset with an even number of values, the
94 // median may not even exist in the dataset, while a quantile value must exist in the
95 // dataset. Note when calculating quantile values, a dataset that does not fall in
96 // specified dataset ranges, is not included via a stride specification, is masked, or
97 // has a weight of zero is not considered a member of the dataset for the pruposes of
98 // quantile calculations.
